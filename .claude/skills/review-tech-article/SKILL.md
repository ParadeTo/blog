# Review 技术文章

## 概述

系统性 review 技术博客文章，覆盖五个维度，以编号列表呈现问题，等待用户选择后再进行修改。**不得在用户确认前擅自修改文件。**

## 工作流程

```
1. 读取文章文件
2. 按五个维度逐一分析
3. 输出优先级摘要（置顶）+ 详细问题列表
4. 等待用户选择编号
5. 仅修改用户选中的条目
```

## 第一步：读取文章

用 Read 工具读取用户指定的文章文件。

## 第二步：按五个维度分析

必须覆盖以下全部五个维度，每个维度单独成节：

| 维度 | 检查重点 |
|---|---|
| **技术准确性** | 代码示例是否正确、概念是否有误、术语是否准确、数据/版本号是否会过时 |
| **表达清晰度** | 句子是否易读、专业术语是否有解释、示例是否直观 |
| **结构完整性** | 章节是否完整、逻辑是否循序渐进、是否缺少关键内容 |
| **语言风格** | 语气是否一致、是否符合目标读者、用词是否自然、配图风格是否与系列文章统一 |
| **逻辑合理性** | 论证是否有支撑、前后是否矛盾、结论是否过于武断 |

## 第三步：输出格式

**必须按以下格式输出，不得省略任何部分：**

### 格式模板

```
## Review 结果

### 优先关注（置顶摘要）
| 编号 | 问题简述 | 维度 | 优先级 |
|---|---|---|---|
| 1 | xxx | 技术准确性 | 高 |
| 3 | xxx | 逻辑合理性 | 中 |

---

### 技术准确性
**问题 1 — [简短标题]**
[说明问题所在，引用原文或代码]
建议修改：[具体修改方案]

（其他维度同理）

---

**请告诉我你想修改哪几条（输入编号即可，例如：1 3）。没有问题的维度可以不列出。**
```

### 输出规则

- 优先级摘要**必须在最前面**，让用户先看到高优问题
- 每个问题必须有编号，整篇文章统一编号（不按维度分别编号）
- 没有发现问题的维度：写"暂无明显问题"，不要省略该维度
- 最后**必须**以"请告诉我你想修改哪几条"结束，不得自行决定修改什么

## 第四步：等待用户选择

收到用户的编号后，逐条修改文章文件，修改完每条后简要说明改了什么。

## 常见错误

| 错误 | 修正 |
|---|---|
| 未覆盖全部五个维度 | 五个维度必须逐一分析，哪怕没有问题也要写"暂无明显问题" |
| 优先级摘要放在末尾 | 必须置顶，让用户先看到最重要的问题 |
| 没有明确邀请用户选择 | 最后一行必须是"请告诉我你想修改哪几条" |
| 分析完直接修改文件 | 必须等用户选择编号后才能修改 |
| 全篇只列问题不给建议 | 每个问题必须附上"建议修改"方案 |
| 忽略配图风格一致性 | 同一系列文章的架构图/流程图应保持统一的配色、形状和排版风格（参考 `source/_posts/ai-agent-skill/flow.png` 的视觉基准：浅灰背景、蓝色圆角矩形、绿色胶囊入口、粉色菱形判断、橙色虚线旁注） |
| 连接线穿越框体 | 流程图/架构图的连接线不得穿越任何节点框体，必须沿框体外侧绕行；线上的标注文字不得覆盖任何框体 |
