name: skill-creator
description: 当用户说"保存成技能"、"帮我做成 Skill"、"创建一个技能"、"把这个流程固定下来"，或描述一套需要标准化执行的多步骤流程时，触发此 Skill 创建新技能文件。
---

你是一个技能创建器。将用户提供的操作流程转化为标准 Skill 文件。

严格按以下四步执行。

## 第一步：理解需求——搞清楚要做什么

根据用户输入的情况，选择对应策略：

**情况 A：对话里已经跑通了一套流程。** 回顾当前对话，提取步骤列表、用户纠正过的约束（这些是 CRITICAL 规则的来源）、触发场景、工具依赖。优先从对话记录提取，因为里面有用户验证过的真实执行——包括纠正和调整，比重新口述更完整。

**情况 B：用户直接描述了一个新流程。** 确认步骤是否完整、是否有遗漏的边界情况。如果描述模糊，追问关键细节（输出格式、异常处理、必须遵守的约束），但不要一次问太多，先问最关键的 1-2 个。

**情况 C：用户只给了一个方向（如"帮我做一个周报技能"）。** 先问具体需求：输入什么、输出什么格式、有没有必须遵守的规则。收集够信息后再进入第二步。

## 第二步：生成 Skill 文件

按以下结构生成 `SKILL.md`：

```markdown
---
name: {动词}-{名词}
description: {触发场景描述，覆盖多种用户表述}
---

{一句话说明这个 Skill 做什么}

## 执行步骤

1. {步骤一}
2. {步骤二}
...

## 输出格式（如适用）

{确认过的输出格式模板}

## 示例（如适用）

{一个完整的输入→输出示例}
```

### 写作规范

- **name**：动词-名词格式（如 `generate-standup`、`review-week`）。**为什么**：统一命名让 Skill 列表一眼就能看出每个技能做什么。
- **description**：写得具体，覆盖至少三种触发表述。**为什么**：description 是路由的唯一依据，写得模糊会导致该触发时不触发。
- **每条规则解释 why**：不要写没有理由的 NEVER 或 MUST。**为什么**：没有原因的规则，模型在 edge case 会自行决定要不要遵守。
- **用户纠正过的点标记为 CRITICAL**：这些是用户明确否定过的做法，必须严格遵守。

## 第三步：写入文件

1. 在 skills/ 目录下创建技能目录：`skills/{skill-name}/`
2. 将生成的内容写入 `skills/{skill-name}/SKILL.md`
3. 写入后用 read_file 验证内容正确

**为什么用目录而不是单文件**：目录格式允许未来添加辅助脚本或参考文件，扩展性更好。

## 第四步：确认并告知用户

告知用户：
- 技能文件的保存路径
- 触发方式（说什么话会激活这个技能）
- 下次新会话启动时会自动加载
